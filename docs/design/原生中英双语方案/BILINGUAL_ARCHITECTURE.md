# BlockType 双语支持架构设计

## 概述

BlockType 编译器支持中英文双语编程，允许开发者在同一项目中自由混用中英文关键字、标识符和注释。本设计文档描述了双语支持的整体架构和关键技术决策。

## 设计目标

### 核心目标

1. **无缝混用**：允许在同一源文件中混用中英文关键字和标识符
2. **标准兼容**：生成的目标代码与标准 C++ 完全兼容
3. **用户友好**：提供双语诊断信息，降低学习和使用门槛
4. **性能优先**：双语支持不应显著影响编译性能

### 非目标

1. 不支持其他语言（如日文、韩文）的关键字
2. 不修改标准 C++ 语义
3. 不提供运行时国际化支持

## 关键设计决策

### 1. 语言混用策略

**决策：支持完全混用模式**

```cpp
// ✅ 允许：同一文件中混用中英文关键字
如果 (x > 0) {          // 中文关键字
    return x;
} else {                // 英文关键字
    return -x;
}

// ✅ 允许：纯中文模式
整数 主函数() {
    整数 变量 = 10;
    返回 变量;
}

// ✅ 允许：纯英文模式（标准 C++）
int main() {
    int variable = 10;
    return variable;
}
```

**编译选项：**
```bash
# 默认：双语混用模式
blocktype source.cpp

# 纯英文模式（禁用中文关键字）
blocktype --lang=en-US source.cpp

# 纯中文模式（禁用英文关键字）
blocktype --lang=zh-CN source.cpp
```

**理由：**
- 最大灵活性：开发者可以根据需要选择语言
- 降低迁移成本：可以逐步将现有代码迁移到中文
- 兼容性：纯英文模式完全兼容标准 C++

### 2. 标识符命名规范

**决策：支持 Unicode 标识符（UAX #31）**

```cpp
// ✅ 允许：中文标识符
整数 计数器 = 0;
字符串 用户名 = "张三";
空 打印信息(字符串 信息) { /* ... */ }

// ✅ 允许：英文标识符
int counter = 0;
std::string username = "John";
void printInfo(std::string info) { /* ... */ }

// ✅ 允许：混合标识符
int 用户数量 = 100;
std::string userName_用户名 = "test";
```

**标识符规则（符合 UAX #31）：**
- 起始字符：`ID_Start` 类别（包括字母、CJK 字符等）
- 后续字符：`ID_Continue` 类别（包括数字、连接符等）
- 规范化：NFC（Canonical Decomposition, followed by Canonical Composition）

**理由：**
- 国际标准：UAX #31 是 Unicode 标准的一部分
- 兼容性好：主流编译器（GCC、Clang）已支持
- 自然直观：使用母语命名更符合思维习惯

### 3. 标准库 API

**决策：标准库保持英文，提供可选中文别名头文件**

```cpp
// 方案 A：使用英文标准库（推荐）
#include <iostream>
#include <vector>
#include <string>

// 方案 B：可选的中文别名头文件
#include <标准输入输出>
#include <向量>
#include <字符串>

// 中文别名示例
namespace std {
    namespace 中文 {
        template<typename T> using 向量 = vector<T>;
        using 字符串 = string;
        using 整数 = int;
        using 浮点数 = double;
    }
}

// 使用示例
std::中文::向量<int> 数据;
std::中文::字符串 名称;
```

**理由：**
- 兼容性优先：保持与现有 C++ 代码的兼容性
- 渐进式本地化：开发者可以选择是否使用中文别名
- 标准遵循：不修改标准库，避免维护负担

### 4. 预处理指令

**决策：支持双语预处理指令**

```cpp
// 英文预处理指令
#include <iostream>
#define MAX_SIZE 100

// 中文预处理指令
#包含 <iostream>
#定义 最大大小 100
```

**理由：**
- 完整的语言体验：预处理指令也是语言的一部分
- 一致性：与关键字双语支持保持一致

### 5. 源文件编码

**决策：UTF-8 优先，支持多种编码**

**支持的编码格式：**
- UTF-8（推荐，默认）
- UTF-8 with BOM
- UTF-16 LE/BE
- UTF-32 LE/BE

**编译器行为：**
```cpp
// 1. 自动检测源文件编码
// 2. 内部统一转换为 UTF-8
// 3. 诊断信息使用用户指定的语言
```

**理由：**
- 现代标准：UTF-8 是现代文本编码的事实标准
- 工具链支持：主流编辑器和工具都支持 UTF-8
- 兼容性：支持多种编码以兼容旧代码

### 6. 诊断消息

**决策：运行时语言切换**

```cpp
// 英文诊断
test.cpp:5:10: error: use of undeclared identifier 'foo'
    foo();
    ^

// 中文诊断
test.cpp:5:10: 错误：使用了未声明的标识符 'foo'
    foo();
    ^
```

**实现方式：**
- 使用 YAML 格式存储诊断消息目录
- 运行时根据 `--lang` 选项加载对应语言的消息
- 支持参数化消息（如 `'%0'`, `'%1'`）

**理由：**
- 用户友好：使用母语显示错误信息更易理解
- 灵活性：开发者可以选择诊断语言
- 易于维护：消息目录独立于代码

### 7. 目标码兼容性

**决策：生成与标准 C++ 完全兼容的目标码**

```cpp
// 源代码（中文）
整数 主函数() {
    整数 计数器 = 0;
    返回 计数器;
}

// 生成的 LLVM IR（与英文版本相同）
define i32 @main() {
  %1 = alloca i32
  store i32 0, i32* %1
  %2 = load i32, i32* %1
  ret i32 %2
}
```

**名称修饰策略：**
- Unicode 标识符使用 UTF-8 编码后进行名称修饰
- 保证与标准 C++ 的 ABI 兼容性

**理由：**
- 互操作性：可以与现有 C++ 库链接
- 生态系统兼容：可以使用现有的工具链和调试器
- 标准遵循：不破坏 C++ 的二进制兼容性

## 模块设计

### 1. Lexer 模块

**职责：**
- Unicode 标识符识别
- 中英文关键字识别
- NFC 规范化
- 源文件编码检测

**关键接口：**
```cpp
class Lexer {
public:
  // 判断是否是标识符起始字符
  bool isIdentifierStart(uint32_t CP);
  
  // 判断是否是标识符后续字符
  bool isIdentifierContinue(uint32_t CP);
  
  // 识别关键字（支持中英文）
  TokenKind recognizeKeyword(StringRef Text);
  
  // NFC 规范化
  std::string normalizeNFC(StringRef Input);
};
```

### 2. Parser 模块

**职责：**
- 双语语法支持
- AST 节点记录源语言

**关键接口：**
```cpp
class Parser {
public:
  // 解析声明（支持双语关键字）
  DeclResult parseDeclaration();
  
  // 解析表达式
  ExprResult parseExpression();
  
  // 解析语句
  StmtResult parseStatement();
};
```

### 3. Semantic Analysis 模块

**职责：**
- 双语诊断信息
- Unicode 标识符符号表

**关键接口：**
```cpp
class Sema {
public:
  // 报告诊断（支持双语）
  void diagnose(SourceLocation Loc, unsigned DiagID, 
                ArrayRef<StringRef> Args = {});
  
  // 名称查找（支持 Unicode 标识符）
  Decl* lookupName(IdentifierInfo *Name, Scope *S);
};
```

### 4. Code Generation 模块

**职责：**
- Unicode 标识符名称修饰
- 生成标准兼容的目标码

**关键接口：**
```cpp
class CodeGenModule {
public:
  // 获取 mangled 名称
  std::string getMangledName(Decl *D);
  
  // 生成函数
  llvm::Function* generateFunction(FunctionDecl *FD);
};
```

## 性能考虑

### 1. 关键字查找

**优化策略：**
- 使用完美哈希表存储关键字
- 中英文关键字使用同一个哈希表
- 查找时间复杂度：O(1)

### 2. Unicode 处理

**优化策略：**
- 使用 ICU 库进行 Unicode 处理
- 缓存 NFC 规范化结果
- 延迟规范化（仅在需要时进行）

### 3. 诊断消息

**优化策略：**
- 延迟加载消息目录
- 缓存格式化后的消息
- 使用字符串池减少内存分配

## 风险与缓解

### 风险 1：性能影响

**风险描述：** Unicode 处理可能影响编译性能

**缓解措施：**
- 使用高效的 ICU 库
- 实现缓存机制
- 延迟处理策略

### 风险 2：兼容性问题

**风险描述：** 生成的目标码可能与标准 C++ 不兼容

**缓解措施：**
- 严格遵循 Itanium C++ ABI
- 全面的兼容性测试
- 与主流编译器进行对比测试

### 风险 3：工具链支持

**风险描述：** 现有工具链可能不支持 Unicode 标识符

**缓解措施：**
- 提供名称修饰映射文件
- 开发调试器扩展
- 提供工具链集成指南

## 总结

BlockType 的双语支持设计遵循以下原则：

1. **用户友好**：使用母语编程更自然直观
2. **标准兼容**：生成的目标码与标准 C++ 完全兼容
3. **性能优先**：通过优化策略确保编译性能
4. **渐进式采用**：支持混用模式，降低迁移成本

通过这些设计，BlockType 将成为一款真正支持中文编程的现代 C++ 编译器。
