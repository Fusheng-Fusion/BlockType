### 1.1.2 各层职责边界

| 层次 | 职责 | 输入 | 输出 | 核心接口 |
|------|------|------|------|----------|
| **前端层** | 源代码 → AST → BTIR | 源代码+LangOpts | `unique_ptr<ir::IRModule>` | `FrontendBase::compile()` |
| **IR 中间层** | 类型/值/指令的统一表示 | 前端产出的IR | 后端可消费的IR | `IRModule`, `IRBuilder`, `IRPass` |
| **后端层** | BTIR → 目标代码 | `const IRModule&`+TargetOpts | 目标文件/汇编 | `BackendBase::emitObject()` |

### 1.1.3 解耦策略

1. **依赖方向单向化**：`Frontend → BTIR ← Backend`，前端和后端之间零依赖
2. **接口抽象**：前端通过 `ir::IRBuilder` 构建 IR；后端通过 `IRConsumer` 遍历 `ir::IRModule` 消费 IR
3. **类型系统独立**：BTIR 拥有自有的类型系统（`ir::IRType`），与 `llvm::Type` 和 `QualType` 均独立
4. **值系统独立**：BTIR 拥有自有的值系统（`ir::IRValue`+`Use`/`User`），与 `llvm::Value` 独立
5. **布局信息独立**：`TargetLayout` 不依赖 LLVM `DataLayout`，IR层可独立计算类型大小
6. **IR层作为独立库**：`libblocktype-ir` 不链接 LLVM，仅依赖 `blocktype-basic`

### 1.1.4 自由组合机制

```bash
bt --frontend=cpp --backend=llvm -O2 source.cpp -o source.o
bt --frontend=cpp --backend=cranelift source.cpp -o source.o
bt --frontend=rust --backend=llvm source.rs -o source.o
bt --frontend=cpp --emit-btir source.cpp -o source.btir
bt --backend=llvm source.btir -o source.o
bt --frontend=cpp --verify-btir source.cpp
```

## 1.2 中间表示（IR）层设计

### 1.2.1 核心决策：自定义 BTIR

| 方案 | 优势 | 劣势 | 评分 |
|------|------|------|------|
| **自定义IR(BTIR)** | 完全可控、轻量、无外部依赖、精确匹配语义 | 需自建类型系统/验证器 | ★★★★★ |
| MLIR | 成熟框架、多级Dialect | 重度依赖LLVM、与解耦目标矛盾 | ★★ |
| LLVM IR | 无需额外实现 | 与LLVM硬耦合，违反R6 | ★ |
| Cranelift IR | 轻量 | Rust生态绑定、C++语义支持弱 | ★★ |

**选择：自定义IR（BTIR）**。理由：R6合规、R1合规、轻量渐进、已有先例（GCC GIMPLE/Rust MIR/Swift SIL）。

### 1.2.2 IR 类型系统

```

├── IRVoidType
├── IRIntegerType     (BitWidth: 1/8/16/32/64/128)
├── IRFloatType       (BitWidth: 16/32/64/80/128)
├── IRPointerType     (PointeeType*, AddressSpace)
├── IRArrayType       (NumElements, ElementType*)
├── IRStructType      (Name, FieldTypes[], IsPacked)
```

`IRTypeContext`：类型工厂+缓存，使用 `DenseMap`/`FoldingSet` 实现类型唯一化，拥有所有 `IRType` 实例生命周期。

**与 `llvm::Type` 的映射**：`LLVMTypeMapper::mapType(ir::IRType*) → llvm::Type*`（后端适配层）

### 1.2.13 IR 内存管理策略

**设计决策**：BumpPtrAllocator + unique_ptr 混合策略

- IR 层与 AST 层同属编译器内部数据结构，生命周期与编译过程绑定
- `ASTContext` 已验证 `BumpPtrAllocator` 的高效性（O(1) 分配，批量释放）
- IR Module 作为顶层对象用 `unique_ptr` 拥有（与 `CodeGenModule` 模式一致）
- IR 内部节点（IRInstruction/IRValue/IRType）用 BumpPtrAllocator 分配

```cpp
namespace blocktype::ir {

class IRContext {
  /// IR 节点分配器（与 ASTContext::Allocator 同类型）
  llvm::BumpPtrAllocator Allocator;

  /// 非平凡资源清理回调（遵循 Clang/ASTContext 模式）
  std::vector<std::function<void()>> Cleanups;

  /// 类型唯一化上下文（拥有所有 IRType 实例）
  IRTypeContext TypeCtx;

public:
  /// 分配 IR 节点内存
  template <typename T, typename... Args>
  T *create(Args &&...args) {
    void *Mem = Allocator.Allocate(sizeof(T), alignof(T));
    T *Node = new (Mem) T(std::forward<Args>(args)...);
    return Node;
  }

  /// 注册非平凡资源清理
  void addCleanup(std::function<void()> Callback) {
    Cleanups.push_back(std::move(Callback));
  }

  /// 保存字符串到 IR 池
  llvm::StringRef saveString(llvm::StringRef Str);

  IRTypeContext &getTypeContext() { return TypeCtx; }

  /// 统计
  size_t getMemoryUsage() const { return Allocator.getTotalMemory(); }
};

} // namespace blocktype::ir
```

**内存管理职责划分**：

| 对象 | 分配方式 | 生命周期 | 拥有者 |
|------|---------|---------|--------|
| `IRContext` | 栈/unique_ptr | 编译过程 | `CompilerInstance` |
| `IRType` 及子类 | `IRContext::create<T>()` (BumpPtrAllocator) | 随 IRContext 销毁 | `IRTypeContext`（缓存+唯一化） |
| `IRValue`/`IRInstruction` | `IRContext::create<T>()` (BumpPtrAllocator) | 随 IRContext 销毁 | `IRBasicBlock`（指令链表） |
| `IRModule` | `unique_ptr<IRModule>` | 编译过程 | `CompilerInstance` |
| `IRFunction`/`IRBasicBlock` | `unique_ptr`（Module 内部容器） | 随 IRModule 销毁 | `IRModule` |
| `IRGlobalVariable` | `unique_ptr`（Module 内部容器） | 随 IRModule 销毁 | `IRModule` |

**与现有代码的对接**：
- `CompilerInstance` 新增 `unique_ptr<ir::IRContext> IRCtx` 成员
- `ASTToIRConverter` 持有 `ir::IRContext&` 引用，通过 `IRCtx.create<T>()` 分配 IR 节点
- `IRToLLVMConverter` 持有 `const ir::IRModule&` 引用，只读遍历

### 1.2.14 IR 线程安全设计

**设计决策**：初期单线程 + 预留接口

- 代码库核心编译路径**零**线程安全设施，IR 层不应过度设计
- AI 模块的并发模式（mutex + lock_guard）可作为未来参考
- `CompilerInvocation` 已设计为 copyable（"Be copyable for multi-threaded compilation"），预留了多实例编译可能

### 1.2.15 IR 错误恢复机制

**设计决策**：复用 DiagnosticsEngine + IR 专用错误结果类型

- 现有 `DiagnosticsEngine` 已有完整的错误报告体系（SourceLocation、DiagLevel、FixItHint）
- `Sema` 层已有 `ExprResult`/`StmtResult` 错误结果模式可参考
- `llvm::Expected<T>` 已在 AI 模块使用，可作为 IR 层错误类型

```cpp
namespace blocktype::ir {

/// IR 转换结果（参考 Sema 的 ExprResult/StmtResult 模式）
class IRConversionResult {
  std::unique_ptr<IRModule> Module;
  bool Invalid = false;
  unsigned NumErrors = 0;
public:
  IRConversionResult() = default;
  IRConversionResult(std::unique_ptr<IRModule> M) : Module(std::move(M)) {}

  static IRConversionResult getInvalid(unsigned Errors = 1) {
    IRConversionResult R;
    R.Invalid = true;
    R.NumErrors = Errors;
    return R;
  }

  bool isInvalid() const { return Invalid; }
  bool isUsable() const { return Module != nullptr && !Invalid; }
  std::unique_ptr<IRModule> takeModule() { return std::move(Module); }
  unsigned getNumErrors() const { return NumErrors; }
};

/// IR 验证结果
class IRVerificationResult {
  bool IsValid;
  SmallVector<std::string, 8> Violations;
public:
  explicit IRVerificationResult(bool Valid) : IsValid(Valid) {}
  void addViolation(const std::string &Msg) { Violations.push_back(Msg); IsValid = false; }
  bool isValid() const { return IsValid; }
  const SmallVector<std::string, 8> &getViolations() const { return Violations; }
};

} // namespace blocktype::ir
```

**错误恢复策略**：

| 阶段 | 错误类型 | 恢复策略 |
|------|---------|---------|
| AST → IR 转换 | 类型映射失败 | 发出 Diag，用 `IROpaqueType` 占位，继续转换 |
| AST → IR 转换 | 语句转换失败 | 发出 Diag，跳过该语句，继续下一语句 |
| AST → IR 转换 | 表达式转换失败 | 发出 Diag，用 `IRConstantUndef` 占位，继续 |
| IR 验证 | 类型完整性/SSA 性质 | 收集所有违规，一次性报告 |
| IR → LLVM 转换 | 不支持的 IR 特性 | 发出 Diag，中止该函数转换，继续下一函数 |

### 1.2.16 IR 版本兼容性

**设计决策**：语义版本 + 二进制格式版本号

```cpp
namespace blocktype::ir {

/// IR 格式版本（独立于编译器版本，允许 IR 格式独立演进）
struct IRFormatVersion {
  uint16_t Major;  /// 主版本：不兼容变更
  uint16_t Minor;  /// 次版本：新增特性，旧读取器可忽略
  uint16_t Patch;  /// 补丁版本：Bug 修复，完全兼容

  static constexpr IRFormatVersion Current() { return {1, 0, 0}; }

  bool isCompatibleWith(IRFormatVersion ReaderVersion) const {
    if (Major != ReaderVersion.Major) return false;
    if (ReaderVersion.Minor < Minor) return false;
    return true;
  }
};

/// IR 二进制文件头
struct IRFileHeader {
  char Magic[4] = {'B', 'T', 'I', 'R'};  /// 魔数 "BTIR"
  IRFormatVersion Version = IRFormatVersion::Current();
  uint32_t Flags = 0;
  uint32_t ModuleOffset = 0;
  uint32_t StringTableOffset = 0;
  uint32_t StringTableSize = 0;
};

} // namespace blocktype::ir
```

### 1.2.17 后端能力声明

**设计决策**：枚举特性集 + 运行时查询接口

- 现有 `TargetInfo` 已有 `isX86_64()`/`isAArch64()` 等平台查询

```cpp
namespace blocktype::ir {

/// IR 特性枚举
enum class IRFeature : uint32_t {
  IntegerArithmetic  = 1 << 0,
  FloatArithmetic    = 1 << 1,
  VectorOperations   = 1 << 2,
  AtomicOperations   = 1 << 3,
  ExceptionHandling  = 1 << 4,
  DebugInfo          = 1 << 5,
  VarArg             = 1 << 6,
  SeparateFloatInt   = 1 << 7,   /// BTIR 特性
  StructReturn       = 1 << 8,
  DynamicCast        = 1 << 9,
  VirtualDispatch    = 1 << 10,

/// 后端能力声明
class BackendCapability {
  uint32_t SupportedFeatures = 0;
  uint32_t RequiredFeatures = 0;
public:
  void declareFeature(IRFeature F) { SupportedFeatures |= static_cast<uint32_t>(F); }
  bool hasFeature(IRFeature F) const { return (SupportedFeatures & static_cast<uint32_t>(F)) != 0; }

};
```

## 1.7 IR层增强设计概要

> 从09-行业前沿补充优化方案提取各增强子系统的概要设计，仅保留接口定义和关键设计决策。完整代码实现见02-IR层实现等具体技术方案文档。

### 1.7.1 Dialect 分层机制

**设计动机**：当前BTIR是单一层级IR，C++特有语义与通用算术指令混合，后端无法选择性消费特定语义层。

**关键设计决策**：引入轻量Dialect分层（借鉴MLIR但更简洁），5个固定Dialect覆盖所有需求。

**核心接口**：

```cpp
namespace blocktype::ir {

/// IR Dialect 标识符
enum class DialectID : uint8_t {
  Core,      /// bt_core: 通用算术/控制流/内存（所有后端必须支持）
  Cpp,       /// bt_cpp: C++特有语义(vtable/RTTI/dynamic_cast/exception)
  Target,    /// bt_target: 目标特定概念(调用约定/ABI/特性指令)
  Debug,     /// bt_debug: 调试信息(DWARF元数据)
  Metadata,  /// bt_meta: 编译器元数据(优化提示/属性)
};

/// IRInstruction 扩展 Dialect 标识
class IRInstruction : public User {
  // ... 现有成员 ...
  DialectID Dialect;          /// 新增：所属Dialect
public:
  DialectID getDialect() const { return Dialect; }
  bool isCoreDialect() const { return Dialect == DialectID::Core; }
};

/// IRType 扩展 Dialect 标识
class IRType {
  // ... 现有成员 ...
  DialectID Dialect;          /// 新增：所属Dialect
public:
  DialectID getDialect() const { return Dialect; }
};

/// Dialect 能力声明（扩展 BackendCapability）
class DialectCapability {
  uint32_t SupportedDialects = 0;
public:
  void declareDialect(DialectID D);
  bool hasDialect(DialectID D) const;
  bool supportsAll(uint32_t Required) const;
};

} // namespace blocktype::ir
```

**Dialect 分层映射**：

| Dialect | 包含内容 | 消费者 |
|---------|---------|--------|
| `bt_core` | Add/Sub/Mul/Load/Store/Br/Call/Ret/ICmp/FCmp/Phi/Select/转换指令 | 所有后端 |
| `bt_cpp` | Invoke/Resume/LandingPad + C++语义元数据 | LLVM后端(完整), Cranelift后端(降级) |
| `bt_target` | 目标特定intrinsics | 特定后端 |
| `bt_debug` | DICompileUnit/DIType/DISubprogram/DILocation | 支持调试的后端 |
| `bt_meta` | 优化提示(inline always/never, 热点标记) | 可选消费 |

**Dialect Lowering 管线**：

```
前端产出: bt_core + bt_cpp + bt_debug + bt_meta
    ↓ DialectLoweringPass (bt_cpp → bt_core)
    ↓   invoke → call + landingpad布局
    ↓   dynamic_cast → __dynamic_cast函数调用
    ↓   vtable → 全局常量 + 间接调用
    ↓
IR Pipeline: bt_core + bt_debug + bt_meta
    ↓ BackendCapability检查
    ↓   Cranelift不支持bt_cpp → 先Lowering再消费
    ↓   LLVM直接消费所有Dialect
    ↓
后端消费: bt_core + (bt_cpp if supported) + bt_debug + bt_meta
```

**兼容性**：DialectID作为附加字段，现有指令默认属于`bt_core`，不改变现有接口。

### 1.7.2 查询式编译架构

**设计动机**：当前增量编译仅为远期预留接口（`IncrementalCompilationState`），缺乏具体架构设计。

**关键设计决策**：引入查询式编译架构（参考Rustc），为增量编译奠定基础。渐进式引入：Phase F基础查询 → Phase G增量编译 → Phase H自动增量。

**核心接口**：

```cpp
namespace blocktype::query {

/// 查询键：唯一标识一个编译计算
using QueryKey = uint64_t;  // 稳定哈希值

/// 查询结果：带指纹的缓存结果
template <typename T>
class QueryResult {
  T Value;
  uint64_t Fingerprint;   /// 结果指纹（用于增量对比）
  bool IsGreen;           /// 增量标记：绿色=有效，红色=需重算
public:
  const T& get() const { return Value; }
  uint64_t getFingerprint() const { return Fingerprint; }
  bool isGreen() const { return IsGreen; }
  void markGreen() { IsGreen = true; }
  void markRed() { IsGreen = false; }
};

/// 查询上下文：编译器的计算引擎
class QueryContext {
  DenseMap<QueryKey, std::any> Cache;
  DenseMap<QueryKey, SmallVector<QueryKey, 4>> Dependencies;
public:
  /// 执行查询（纯函数语义）
  template <typename ResultT, typename FuncT>
  QueryResult<ResultT> query(QueryKey K, FuncT&& Compute);

  /// 增量编译：尝试标记为绿色（红绿标记算法）
  bool tryMarkGreen(QueryKey K);

  /// 持久化/加载
  bool saveToDisk(StringRef Path);
  bool loadFromDisk(StringRef Path);
};

/// 编译查询定义（渐进式引入）
namespace queries {
QueryResult<std::unique_ptr<ir::IRModule>>
  convertASTToIR(QueryContext& Ctx, const TranslationUnitDecl* TU);
QueryResult<ir::IRVerificationResult>
  verifyIRModule(QueryContext& Ctx, const ir::IRModule* M);
QueryResult<std::unique_ptr<ir::IRModule>>
  optimizeIR(QueryContext& Ctx, const ir::IRModule* M);
} // namespace queries

/// 投影查询模式（依赖防火墙）
QueryResult<ir::IRFunction*>
  getFunction(QueryContext& Ctx, const ir::IRModule* M, StringRef Name);

} // namespace blocktype::query
```

**渐进式路线**：

| 阶段 | 能力 | 依赖 |
|------|------|------|
| Phase A-E | 无查询系统，直接调用 | — |
| Phase F | 引入QueryContext + 基础查询 | IR层完成 |
| Phase G | 增量编译：红绿标记 + 缓存持久化 | 依赖图 + 指纹 |
| Phase H | 自动增量：文件监控 + 智能重编译 | Phase G |

### 1.7.3 编译器可观测性

**设计动机**：当前编译器无任何性能遥测或自剖析能力，编译慢时无法定位瓶颈。

**关键设计决策**：引入编译器自剖析框架（参考LLVM `-ftime-report` 和 Rustc `-Zself-profile`），通过RAII自动记录编译阶段耗时。

**核心接口**：

```cpp
namespace blocktype::telemetry {

/// 编译阶段枚举
enum class CompilationPhase : uint8_t {
  Frontend, IRGeneration, IRValidation,
  IROptimization, BackendCodegen, BackendOptimize,
  CodeEmission, Linking,
};

/// 编译事件记录
struct CompilationEvent {
  CompilationPhase Phase;
  std::string Detail;
  uint64_t StartNs, EndNs;
  size_t MemoryBefore, MemoryAfter;
  bool Success;
};

/// 遥测收集器（编译器内置）
class TelemetryCollector {
  bool Enabled = false;
  SmallVector<CompilationEvent, 64> Events;
public:
  void enable() { Enabled = true; }

  /// RAII阶段守卫（析构时自动记录）
  class PhaseGuard {
    TelemetryCollector& Collector;
    CompilationPhase Phase;
    uint64_t StartNs;
  public:
    PhaseGuard(TelemetryCollector& C, CompilationPhase P, StringRef D);
    ~PhaseGuard();
  };

  PhaseGuard scopePhase(CompilationPhase P, StringRef Detail = "");
```

### 1.7.4 编译器安全架构

**设计动机**：当前方案无安全相关设计，编译器作为关键基础设施需要IR完整性保证、编译过程可重现性、IR Pass正确性验证。

**关键设计决策**：引入三层安全机制——IR完整性验证、可重现构建、Pass不变量检查。

**核心接口**：


## 1.8 行业标准对标

> 将BlockType编译器与行业主流编译器进行系统性对标，明确各特性的对标程度和关键设计决策。

### 1.8.1 LLVM/Clang 对标

| 特性 | LLVM/Clang 实现 | BlockType 对标方案 | 对标程度 |
|------|----------------|-------------------|---------|
| 多级IR | LLVM IR (单一层级) + MIR (机器IR) | BTIR Dialect分层(bt_core/bt_cpp/bt_target) | **超越**：Dialect机制比LLVM单一IR更灵活 |
| Pass管线 | New Pass Manager (模块化Pass管线) | IR PassManager + DialectLoweringPass | **对齐**：Pass管线设计对齐NPM |
| 增量编译 | ThinLTO + 增量模块编译 | 查询式编译 + 红绿标记 | **超越**：查询式架构比LLVM增量编译更精细 |
| 诊断系统 | Clang Diagnostics (FixIt + 模板差异) | StructuredDiagnostic + FixIt + SARIF | **对齐+增强**：增加SARIF格式 |
| 调试信息 | LLVM DebugInfo Metadata (DWARF5) | IRDebugMetadata + DWARF5 + CodeView | **对齐** |
| 插件系统 | New Pass Manager Plugin | CompilerPlugin + PluginManager | **对齐** |
| 缓存系统 | 无内置缓存 | CompilationCacheManager (本地+远程) | **超越**：LLVM无内置编译缓存 |
| 遥测 | -ftime-report + -ftime-trace | TelemetryCollector + Chrome Trace | **对齐** |
| 模糊测试 | Bugpoint + lit测试 | IRFuzzer + DifferentialTester + IREquivalenceChecker | **超越**：三件套覆盖更全面 |
| AI调优 | MLGO (ML引导内联/寄存器分配) | AIAutoTuner (RL驱动Pass序列) | **超越**：Pass序列级调优比单Pass调优更全面 |

### 1.8.2 MLIR 对标

| 特性 | MLIR 实现 | BlockType 对标方案 | 对标程度 |
|------|----------|-------------------|---------|
| Dialect机制 | 完整Dialect基础设施(定义/验证/变换/降级) | 轻量DialectID + DialectLoweringPass | **部分对齐**：BTIR Dialect是MLIR的轻量化实现 |
| Dialect定义 | TableGen + ODS(Operation Definition Specification) | 手写C++ + DialectID枚举 | **简化**：远期可引入声明式定义 |
| Dialect变换 | PatternRewriter + DialectConversion框架 | DialectLoweringPass | **简化**：BTIR仅做降级，不做Dialect间变换 |
| 多级IR | 无限层级Dialect嵌套 | 5个固定Dialect(bt_core/cpp/target/debug/meta) | **简化**：5层足够，避免过度复杂 |
| 验证 | ODS自动生成验证逻辑 | DialectCapability + BackendCapability检查 | **对齐** |

**关键决策**：BTIR选择轻量Dialect而非完整MLIR Dialect，理由：
1. BlockType编译器规模不需要MLIR级别的Dialect复杂度
2. 轻量Dialect实现成本远低于完整MLIR集成
3. 5个固定Dialect覆盖BlockType所有需求
4. 远期如需更多Dialect，可渐进扩展DialectID枚举

### 1.8.3 GCC 对标

| 特性 | GCC 实现 | BlockType 对标方案 | 对标程度 |
|------|---------|-------------------|---------|
| 多级IR | GENERIC → GIMPLE → RTL → Machine | bt_core → bt_cpp降级 → 后端内部表示 | **部分对齐**：BTIR层级更简洁 |

| LTO | Full LTO + ThinLTO | 远期：IR缓存 + ThinLTO式增量 | **远期对齐** |
| 缓存 | 无内置缓存 | CompilationCacheManager | **超越** |
| 调试信息 | DWARF5 (GCC 11+) | DWARF5 + CodeView | **对齐+增强** |

### 1.8.4 Rustc 对标

| 特性 | Rustc 实现 | BlockType 对标方案 | 对标程度 |
|------|-----------|-------------------|---------|
| 查询式编译 | 完整查询系统(所有编译计算通过查询) | QueryContext + 基础查询(渐进式) | **部分对齐**：远期完整实现 |

### 1.8.5 CompCert 对标

| 特性 | CompCert 实现 | BlockType 对标方案 | 对标程度 |
|------|-------------|-------------------|---------|
| 形式化验证 | Coq证明每Pass语义保持 | PassInvariantChecker + IREquivalenceChecker | **简化对齐**：自动化验证而非Coq证明 |
| 可重现构建 | CompCert本身可重现 | IRModule可重现模式 + SOURCE_DATE_EPOCH | **对齐** |
| 正确性保证 | 数学证明编译正确性 | 差分测试 + 等价性验证 + 不变量检查 | **实用化替代**：无法达到CompCert级别证明，但提供实用验证 |

**关键决策**：BlockType选择实用化验证而非完整形式化验证，理由：
1. CompCert的Coq证明需要专业形式化验证团队，BlockType项目规模不支撑
2. PassInvariantChecker + IREquivalenceChecker + 差分测试可覆盖大部分正确性需求
3. 远期(Phase H)可引入符号执行验证，逐步逼近形式化验证水平

### 1.8.6 Cranelift 对标

| 特性 | Cranelift 实现 | BlockType 对标方案 | 对标程度 |
|------|---------------|-------------------|---------|
| 模块化后端 | 独立指令选择(ISLE) + 寄存器分配 + 代码发射 | InstructionSelector + RegisterAllocator + CodeEmitter | **对齐** |
| 声明式指令选择 | ISLE DSL(完整声明式规则语言) | LoweringRule + .isle规则文件(远期) | **部分对齐**：远期完整实现 |
| 可插拔寄存器分配 | 多种分配策略(Greedy/LinearScan) | RegAllocStrategy(Greedy/Fast/Basic) | **对齐** |
| 安全设计 | Spectre防御 + 可重现构建 | IR沙箱 + 可重现构建 + IR完整性验证 | **对齐+增强** |
| WASM集成 | WASM后端(完整WASM支持) | WASM调用约定 + FFI(远期) | **远期对齐** |
