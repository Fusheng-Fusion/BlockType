# BlockType 多前端多后端编译器架构重构 — 最终融合方案

> 本文档融合方案0（architect，1454行，24个具名Task）和方案1（architect-v2，843行，37个Task）的优势，以红线原则为裁判依据，生成最终方案。
> 文档版本：v1.0-final | 日期：2026-04-24 | 融合者：architect-v3

---

## 红线 Checklist

| # | 红线 | 说明 |
|---|------|------|
| R1 | 架构优先 | 任何决策以架构正确性和灵活性为最高优先级 |
| R2 | 多前端多后端自由组合 | 前端和后端独立替换、自由组合，无硬耦合 |
| R3 | 渐进式改造 | 每阶段可编译、可测试，禁止一次性重写 |
| R4 | 现有功能不退化 | 改造中所有功能保持等价或更优 |
| R5 | 接口抽象优先 | 新增代码通过抽象接口交互 |
| R6 | IR 中间层解耦 | 前端产出 IR，后端消费 IR，前后端不直接交互 |

## 一、架构设计对比

### 1.1 IR 层设计差异

| 维度 | 方案0 | 方案1 | 优势方 |
|------|-------|-------|--------|
| 类型系统命名 | `btir::Type` 层级，`BTIRContext` | `ir::IRType` 层级，`IRTypeContext` | 方案1（命名更清晰，避免与AST Type冲突） |
| 类型唯一化 | `BTIRContext` 用 `DenseMap`/`FoldingSet` | `IRTypeContext` 用 `DenseMap` | 方案0（FoldingSet更高效） |
| 类型大小计算 | `getSizeInBits()` 无参（隐含依赖Context） | `getSizeInBits(const TargetLayout&)` 显式依赖 | **方案1**（R5合规：显式依赖优于隐式） |
| TargetLayout | 无独立TargetLayout，类型大小隐含在BTIRContext | 独立`TargetLayout`，不依赖LLVM DataLayout | **方案1**（R6合规：IR层不依赖LLVM） |
| 值系统 | `Value`+`Use`/`User` use-def chain | `IRValue` 简化设计，无use-def chain | 方案0（use-def chain是IR标准基础设施，支持优化Pass） |
| 指令集 | 分类简洁（终结/算术/内存/转换/调用/聚合/元信息） | 分类更细（终结/整数二元/浮点二元/位运算/内存/转换/比较/调用/其他/调试） | **方案1**（整数/浮点分离更利于后端映射） |
| 模块结构 | Module含Functions/Globals/Aliases | IRModule含Functions/Globals/Metadata+IRFunctionDecl | **方案1**（IRFunctionDecl区分声明/定义更精确） |
| IR构建器 | 含常量工厂+指令创建 | 仅指令创建 | 方案0（集成常量工厂更便捷） |
| IR验证器 | `VerifierPass` 继承 `Pass` | 独立验证规则列表 | 方案0（可纳入PassManager管线） |
| IR序列化 | 文本+二进制 | 仅文本格式 | 方案0（更完整） |
| IR文本格式示例 | 无 | 有完整示例 | 方案1（有助于实现） |
| IR Pass框架 | `Pass`+`PassManager` 完整框架 | 无 | **方案0**（IR层基础设施必需） |
| IR调试信息 | DebugMetadata层级设计（DICompileUnit/DIType/DISubprogram/DILocation） | 仅提及 | **方案0**（更具体可实施） |
| IRConsumer | 有`IRConsumer`接口供后端遍历 | 无 | 方案0（后端遍历IR的标准接口） |

### 1.2 前端抽象层设计差异

| 维度 | 方案0 | 方案1 | 优势方 |
|------|-------|-------|--------|
| FrontendBase接口 | `compile(StringRef)` + `compileFromString()` + `canHandle()` | `compile(Filename, IRTypeContext&, TargetLayout&)` | **方案1**（显式传入更符合R5/R6） |
| FrontendRegistry | 含`autoSelect()` + `ExtToName`扩展名映射 | 仅register+create | **方案0**（autoSelect更实用） |
| ASTToIRConverter | 子组件拆分(IREmitExpr/Stmt/CXX/ConstEmitter) | 单一converter | **方案0**（符合R5接口抽象） |

### 1.3 后端抽象层设计差异

| 维度 | 方案0 | 方案1 | 优势方 |
|------|-------|-------|--------|
| BackendBase接口 | emitObject+emitAssembly+canHandle+optimize | emitObject+emitAssembly+emitIRText | **方案0**（更完整）+方案1的emitIRText |
| 类型映射规则 | 无显式映射表 | 有完整映射表 | **方案1**（实现直接参考） |
| VTable/RTTI | LLVM后端内部处理 | 显式Task C.8 | **方案1**（确保不遗漏） |
| 调试信息转换 | DebugMetadata→llvm::DI*映射 | IRDebugInfoBuilder→DWARF | 方案0（更具体） |

### 1.4 编译管线编排差异

| 维度 | 方案0 | 方案1 | 优势方 |
|------|-------|-------|--------|
| CompilerInstance | 具体设计（5步管线） | 简要描述 | 方案0 |
| 命令行选项 | 6个选项（含--emit-btir-bitcode/--btir-only/--load-btir） | 4个选项（含--verify-btir） | **方案0**（更完整）+方案1的--verify-btir |
| 自动注册 | 有CppFrontendRegistration/LLVMBackendRegistration | 无 | **方案0**（更优雅） |

## 二、技术方案对比

### 2.1 IR 层实现差异

| 维度 | 方案0 | 方案1 | 优势方 |
|------|-------|-------|--------|
| 命名空间 | `btir::` | `blocktype::ir::` | **方案1**（嵌套命名空间更规范） |
| CMake | libblocktype-ir不链接LLVM | 显式声明仅依赖blocktype-basic | **方案1**（更规范） |

### 2.2 前端适配策略差异

| 维度 | 方案0 | 方案1 | 优势方 |
|------|-------|-------|--------|
| CGCXX拆分 | 7子文件(Layout/Ctor/Dtor/VTable/RTTI/Inherit/Thunk) | 6子文件(Layout/Ctor/Dtor/VTable/RTTI/Misc) | **方案0**（Inherit/Thunk分离更清晰） |
| Mangler | `Mangler(ASTContext&, const TargetInfo&)` | `IRMangler(ASTContext&, TargetLayout&)` | **方案1**（依赖IR层TargetLayout更符合R6） |
| TargetInfo/ABIInfo | 移至Target/目录 | 布局→IR层，ABI→后端层 | **方案1**（更符合R6） |
| ABIArgInfo | 无提及 | `llvm::Type*`→`ir::IRType*` | **方案1**（显式处理LLVM类型解耦） |
| 共存策略 | 无显式策略 | 三阶段：共存→迁移→替换 | **方案1**（更符合R3） |
| 重构映射表 | 有完整映射表 | 有三阶段策略+组件映射表 | **方案0**（映射表更完整）+方案1（三阶段更渐进） |

### 2.3 后端适配策略差异

| 维度 | 方案0 | 方案1 | 优势方 |
|------|-------|-------|--------|
| IRToLLVMConverter | 3天整体实现 | 拆分为类型(2天)+指令(3天)+模块(2天) | **方案1**（更渐进） |
| 优化Pass | IR层Pass抽象框架 | 初期直接用LLVM Pass | 互补（取方案0框架+方案1务实策略） |

## 四、各自独特优势

### 4.1 方案0 有而方案1 没有的内容

1. **Use/User use-def chain**：IR标准基础设施，支持优化Pass
2. **IRConsumer 接口**：后端遍历IR的标准接口
3. **Pass+PassManager+VerifierPass 框架**：IR层Pass基础设施
4. **DebugMetadata 层级设计**：DICompileUnit/DIType/DISubprogram/DILocation
5. **IRWriter/IRReader 二进制格式**：完整序列化支持
6. **FrontendRegistry::autoSelect()**：基于文件扩展名自动选择前端
7. **自动注册机制**：CppFrontendRegistration/LLVMBackendRegistration
8. **完整命令行选项**：--emit-btir-bitcode/--btir-only/--load-btir
9. **Phase B/C 并行策略**：缩短工期
10. **性能量化数据**：编译速度/内存/二进制大小的具体影响数字
11. **Phase8 交互矩阵**：各组件在各Phase的交互关系
12. **BTIR 指令集完整参考**：所有指令的文本格式语法
13. **IRBuilder 集成常量工厂**：便捷的常量创建

### 4.2 方案1 有而方案0 没有的内容

1. **TargetLayout 独立设计**：IR层不依赖LLVM DataLayout（R6合规关键）
2. **IRType::getSizeInBits(const TargetLayout&)** 显式依赖
3. **IRFunctionDecl** 区分函数声明和定义
4. **FrontendBase::compile() 显式传入 IRTypeContext 和 TargetLayout**
5. **三阶段共存策略**：共存期→迁移期→替换期
6. **IRMangler(ASTContext&, TargetLayout&)** 依赖IR层而非AST层
7. **ABIArgInfo 改造**：`llvm::Type*` → `ir::IRType*`
8. **TargetInfo/ABIInfo 归属策略**：布局→IR层，ABI→后端层
9. **VTable/RTTI 显式Task**：确保不遗漏
10. **IR文本格式完整示例**：实际可参考的实现示例
11. **每个Phase红线对照**：逐条验证合规性
12. **风险系数调整**：工作量估算含风险系数
13. **CMake 显式依赖声明**：libblocktype-ir 仅依赖 blocktype-basic
14. **`blocktype::ir::` 嵌套命名空间**
15. **37个细粒度Task**：更安全、更渐进

### 4.3 两方案共同缺失的内容（已补充完成 ✓）

> 以下 8 项缺失内容已全部补充并融合到各方案文档中。补充方案基于代码库实际扫描结果设计。

1. **IR层内存管理策略** → 已融合：IRContext + BumpPtrAllocator + unique_ptr 混合策略
2. **IR层线程安全设计** → 已融合：初期单线程 + IRThreadingMode 预留接口
3. **错误恢复机制** → 已融合：IRConversionResult + 错误占位策略
4. **IR版本兼容性** → 已融合：IRFormatVersion + IRFileHeader
5. **前端-IR层契约测试** → 已融合：6 项契约不变量 + GTest 测试
6. **后端能力声明** → 已融合：BackendCapability + IRFeature 枚举
7. **IR优化Pass的具体实现** → 已融合：4 个基础 Pass + createDefaultIRPipeline()
8. **增量编译支持** → 已融合：IncrementalCompilationState 远期预留接口
