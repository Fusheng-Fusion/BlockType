## 2.3.4 后端能力声明与检查

### 后端能力声明

```cpp
// LLVM 后端能力
BackendCapability LLVMBackend::getCapability() const {
  BackendCapability Cap;
  Cap.declareFeature(IRFeature::IntegerArithmetic);

  Cap.declareFeature(IRFeature::AtomicOperations);
  Cap.declareFeature(IRFeature::ExceptionHandling);
  Cap.declareFeature(IRFeature::DebugInfo);
  Cap.declareFeature(IRFeature::VarArg);
  Cap.declareFeature(IRFeature::SeparateFloatInt);
  Cap.declareFeature(IRFeature::StructReturn);
  Cap.declareFeature(IRFeature::DynamicCast);
  Cap.declareFeature(IRFeature::VirtualDispatch);
  // Coroutines: 未来支持
  return Cap;
}

// Cranelift 后端能力（未来）
BackendCapability CraneliftBackend::getCapability() const {
  BackendCapability Cap;
  Cap.declareFeature(IRFeature::IntegerArithmetic);
  Cap.declareFeature(IRFeature::FloatArithmetic);
  Cap.declareFeature(IRFeature::VectorOperations);
  // 无 ExceptionHandling、无 DebugInfo、无 DynamicCast
  return Cap;

```

### 后端能力检查集成

```
CompilerInstance::compileFile():
  1. Frontend → IRModule (前端生成全部 IR 特性)
  2. BackendCapability = Backend->getCapability()
  3. RequiredFeatures = IRModule::getRequiredFeatures()
  4. Unsupported = BackendCapability.getUnsupported(RequiredFeatures)
  5. if (Unsupported != 0) → 报告不支持的特性，中止或降级
  6. Backend->emitObject(IRModule)
```

**降级策略**：当后端不支持某些 IR 特性时：
- `ExceptionHandling`：降级为 `call`（无异常表），发出警告
- `DebugInfo`：降级为无调试信息，发出警告
- `DynamicCast`/`VirtualDispatch`：**不可降级**，中止编译
- `VectorOperations`：降级为标量循环，发出警告

## 2.3.5 VTable/RTTI 生成（LLVM 后端专属）

**策略**（取自方案1显式Task）：

VTable 和 RTTI 是 Itanium ABI 特定概念，不属于 IR 层通用语义。在 IR 层中，它们通过**全局常量+函数调用约定**表达。具体的 VTable 布局和 RTTI type_info 结构在 LLVM 后端的 `IRToLLVMConverter` 中生成。

**LLVM 后端 VTable/RTTI 生成流程**：
1. `ASTToIRConverter` 识别 C++ 类需要 VTable/RTTI
2. 在 IR 中创建占位全局变量（`ir::IRGlobalVariable`，类型为 `ir::IROpaqueType`）
3. `IRToLLVMConverter` 在转换时根据 Itanium ABI 填充 VTable/RTTI 的具体布局
4. 虚函数调用在 IR 中为间接调用（通过函数指针），LLVM 后端映射为虚表查找

## 2.3.6 声明式指令选择（ISLE式）

> 来源：09-行业前沿补充优化方案 §2.3

借鉴 Cranelift ISLE（Instruction Selection/Lowering Expressions）架构，引入声明式指令选择机制。当前后端指令选择通过手写C++映射，可维护性差；声明式规则允许以DSL定义指令选择模式，支持模式穷尽性检查，可维护性远高于手写C++。

### 指令选择规则（声明式定义）

```cpp
namespace blocktype::backend {

/// 指令选择规则（声明式定义）
struct LoweringRule {
  ir::Opcode SourceOp;           /// 源IR操作码
  ir::DialectID SourceDialect;  /// 源Dialect
  std::string TargetPattern;    /// 目标模式（DSL表达式）
  std::string Condition;        /// 应用条件（可选）
  int Priority;                 /// 优先级（多规则匹配时选择最高优先级）
};

/// 指令选择器接口
class InstructionSelector {
public:
  virtual ~InstructionSelector() = default;

  /// 为IR指令选择目标指令
  virtual bool select(const ir::IRInstruction& I,
                      TargetInstructionList& Output) = 0;

  /// 加载声明式规则文件
  virtual bool loadRules(StringRef RuleFile) = 0;

  /// 验证规则完整性（穷尽性检查）
  virtual bool verifyCompleteness() = 0;
};

/// 目标指令（后端内部表示）
class TargetInstruction {
  std::string Mnemonic;         /// 助记符（如 "add rax, rbx"）
  SmallVector<unsigned, 4> UsedRegs;   /// 使用的寄存器
  SmallVector<unsigned, 4> DefRegs;    /// 定义的寄存器
  SmallVector<ir::IRValue*, 2> IROperands; /// 关联的IR操作数
public:
  StringRef getMnemonic() const { return Mnemonic; }
};

} // namespace blocktype::backend
```

### .isle 规则文件示例

```
;; BTIR → x86_64 指令选择规则
;; 算术指令
(rule (lower (add i32 %x %y)) (add32 %x %y) :priority 1)
(rule (lower (add i64 %x %y)) (add64 %x %y) :priority 1)
(rule (lower (mul i32 %x %y)) (imul32 %x %y) :priority 1)

;; 常量优化
(rule (lower (add i32 %x (const i32 1))) (inc32 %x) :priority 2)
(rule (lower (add i32 %x (const i32 -1))) (dec32 %x) :priority 2)

;; 内存指令
(rule (lower (load i32 %ptr)) (mov32 (mem %ptr)) :priority 1)
(rule (lower (store i32 %val %ptr)) (mov32 (mem %ptr) %val) :priority 1)
```

### 与 IRToLLVMConverter 的关系

```
现有 IRToLLVMConverter
    ↓ 扩展（Phase D 引入 InstructionSelector 接口）
    ├── Phase A-E：IRToLLVMConverter 直接映射（现有方式不变）
    ├── Phase F：引入 InstructionSelector 接口（LLVM后端内部使用）
    │   IRToLLVMConverter::convertInstruction()
    │       ↓ 委托给 InstructionSelector::select()
    │       ↓ 返回 TargetInstructionList
    │       ↓ IRToLLVMConverter 将 TargetInstruction 映射为 LLVM IR
    └── Phase G+：声明式规则文件(.isle)解析器
        InstructionSelector::loadRules("x86_64.isle")
            ↓ 加载声明式规则
            ↓ 编译为高效匹配代码
            ↓ select() 使用规则匹配
```

**兼容性说明**：
- Phase A-E 仍使用 `IRToLLVMConverter` 直接映射，无行为变化
- Phase F 引入 `InstructionSelector` 接口，作为 `IRToLLVMConverter` 的内部子组件
- 声明式规则文件（`.isle`）为远期目标（Phase F+）

## 2.3.7 可插拔寄存器分配

> 来源：09-行业前沿补充优化方案 §2.3

借鉴 Cranelift regalloc2 的可插拔设计，将寄存器分配设计为可插拔接口，后端可选择最适合的分配策略。

```cpp
namespace blocktype::backend {

/// 寄存器分配策略枚举
enum class RegAllocStrategy {
  Greedy,       /// 贪心算法（高质量，LLVM默认）
  Fast,         /// 线性扫描（高速度，调试构建用）
  Basic,        /// 基础算法（简单正确，验证用）
};

/// 寄存器分配器接口（可插拔）
class RegisterAllocator {
public:
  virtual ~RegisterAllocator() = default;
  virtual StringRef getName() const = 0;
  virtual RegAllocStrategy getStrategy() const = 0;

  /// 执行寄存器分配
  virtual bool allocate(TargetFunction& F,
                        const TargetRegisterInfo& TRI) = 0;
};

/// 寄存器分配器工厂
class RegAllocFactory {
public:
  static std::unique_ptr<RegisterAllocator>
  create(RegAllocStrategy Strategy);
};

} // namespace blocktype::backend
```

**策略选择指南**：

| 策略 | 适用场景 | 质量 | 速度 | 实现阶段 |
|------|---------|------|------|---------|
| `Greedy` | 发布构建（-O2/-O3） | 高 | 中 | Phase D |
| `Fast` | 调试构建（-O0/-O1） | 中 | 快 | Phase D |
| `Basic` | 验证/测试 | 基础 | 快 | Phase D |

**兼容性说明**：
- Phase A-E 寄存器分配完全委托 LLVM（现有方式不变）
- Phase D 引入 `RegisterAllocator` 接口和 `RegAllocFactory`
- Phase D 实现 Greedy/Basic 两种策略（Fast为远期）

## 2.3.8 后端管线模块化

> 来源：09-行业前沿补充优化方案 §2.3

借鉴 Cranelift 的细粒度 crate 模块化架构，将后端内部管线拆分为独立子组件，每个子组件有明确的接口和职责。

### 后端管线模块化拆分

```
BackendBase
├── InstructionSelector    (声明式指令选择)
│   ├── 接口：InstructionSelector
│   ├── 输入：ir::IRInstruction
│   └── 输出：TargetInstructionList
│
├── RegisterAllocator      (可插拔寄存器分配)
│   ├── 接口：RegisterAllocator
│   ├── 输入：TargetFunction + TargetRegisterInfo
│   └── 输出：分配后的 TargetFunction
│
├── CodeEmitter            (代码发射)
│   ├── 接口：CodeEmitter
│   ├── 输入：分配后的 TargetFunction
│   └── 输出：目标代码（ELF/Mach-O/COFF）
│
├── FrameLowering          (栈帧降低)
│   ├── 接口：FrameLowering
│   ├── 输入：TargetFunction + ABI信息
│   └── 输出：含栈帧指令的 TargetFunction
│
├── TargetLowering         (目标特定降低)
│   ├── 接口：TargetLowering
│   ├── 输入：ir::IRInstruction（特定Dialect）
│   └── 输出：目标特定的 TargetInstruction
│
└── DebugInfoEmitter       (调试信息发射)
    ├── 接口：DebugInfoEmitter
    ├── 输入：ir::IRModule + 调试元数据
    └── 输出：DWARF5/CodeView调试段
```

### 后端管线执行流程

```
IRModule 输入
    ↓
IRToLLVMConverter::convert()（Phase A-E 直接映射）
    或
BackendBase::emitObject()（Phase D+ 模块化管线）
    ├── InstructionSelector::select()     → 选择目标指令
    ├── RegisterAllocator::allocate()     → 分配寄存器
    ├── FrameLowering::lower()            → 生成栈帧代码
    ├── TargetLowering::lower()           → 目标特定降低
    ├── DebugInfoEmitter::emit()          → 生成调试信息
    └── CodeEmitter::emit()               → 发射目标代码
```

**兼容性说明**：
- Phase A-E 后端管线为 `IRToLLVMConverter` 直接映射，无模块化拆分
- Phase D 引入各子组件接口定义
- Phase D 实现后端管线模块化拆分（D-F9 任务）

## 2.3.9 调试信息升级 — 后端调试发射

> 来源：09-行业前沿补充优化方案 §2.12

在IR层建立调试元数据体系，后端通过 `DebugInfoEmitter` 接口将IR调试元数据发射为DWARF5或CodeView格式。当前调试信息生成完全委托LLVM，本节定义后端调试信息发射的抽象接口。

### IR调试元数据节点

```cpp
namespace blocktype::ir::debug {

/// IR调试元数据节点（参考LLVM DebugInfo Metadata）
class IRDebugMetadata {
public:
  /// 源位置信息
  struct SourceLocation {
    StringRef File;
    uint32_t Line;
    uint32_t Column;
    uint32_t StartLine;     /// 函数/块起始行
    uint32_t StartColumn;
  };

  /// 类型调试信息
  struct DIType {
    enum class Kind {
      Basic,       /// 基本类型（int, float）
      Pointer,     /// 指针类型
      Array,       /// 数组类型
      Struct,      /// 结构体
      Class,       /// 类（含继承信息）
      Enum,        /// 枚举
      Function,    /// 函数类型
      Template,    /// 模板实例化
    };
    Kind TypeKind;
    StringRef Name;
    uint64_t SizeInBits;
    uint64_t AlignInBits;
    SmallVector<DIType*, 4> Members;     /// 成员类型
    DIType* BaseType;                     /// 基类类型
    SmallVector<DIType*, 2> TemplateArgs; /// 模板参数
  };

  /// 子程序调试信息
  struct DISubprogram {
    StringRef Name;
    StringRef LinkageName;
    DIType* Type;
    SourceLocation DefLoc;
    bool IsDefinition;
    bool IsLocal;
    bool IsOptimized;
  };

  /// 编译单元调试信息
  struct DICompileUnit {
    StringRef SourceFile;
    StringRef Producer;          /// "BlockType Compiler v1.0"
    bool IsOptimized;
    uint32_t DWARFVersion;       /// DWARF版本（默认5）
    StringRef TargetTriple;
  };
};

/// IR指令调试标注（每条IR指令可关联源位置）
class IRInstructionDebugInfo {
  IRDebugMetadata::SourceLocation Loc;
  Optional<IRDebugMetadata::DISubprogram*> Subprogram;
  bool IsArtificial;      /// 编译器生成的指令（无源位置）
  bool IsInlined;         /// 内联展开的指令
  Optional<IRDebugMetadata::SourceLocation> InlinedAt;
};

} // namespace blocktype::ir::debug
```

### DebugInfoEmitter 接口

```cpp
namespace blocktype::ir::debug {

/// IR调试信息发射器接口
class DebugInfoEmitter {
public:
  virtual ~DebugInfoEmitter() = default;

  /// 为后端提供调试信息
  virtual void emitDebugInfo(const ir::IRModule& M,
                              raw_ostream& OS) = 0;

  /// 生成DWARF5调试段
  virtual void emitDWARF5(const ir::IRModule& M,
                           raw_ostream& OS) = 0;

  virtual void emitCodeView(const ir::IRModule& M,
                             raw_ostream& OS) = 0;
};

```

### DWARF5 发射说明

DWARF5 是当前调试信息标准（GCC 11+默认），相比DWARF4的主要改进：
- 分离调试文件（`.dwo`）支持更快的链接
- 类型单元索引加速调试信息查找
- 增量调试信息支持部分更新

**LLVM后端DWARF5发射流程**：
```
IRModule（含 ir::debug::IRDebugMetadata）
    ↓ IRToLLVMConverter 映射调试元数据
    ↓ ir::debug::DICompileUnit → llvm::DICompileUnit
    ↓ ir::debug::DIType → llvm::DIType
    ↓ ir::debug::DISubprogram → llvm::DISubprogram
    ↓ ir::debug::DILocation → llvm::DILocation
    ↓
LLVM TargetMachine::addPassesToEmitFile()
    ↓ 生成DWARF5调试段（.debug_info, .debug_abbrev, .debug_line 等）
    ↓
目标文件（ELF/Mach-O/COFF）
```

### CodeView 发射说明

CodeView 是 Windows/MSVC 平台的调试格式，用于MSVC调试器兼容：
- Phase D 定义 `DebugInfoEmitter` 接口
- Phase E 实现 CodeView 调试信息发射（E-F8 任务）
- Windows 后端选择 CodeView，其他平台选择 DWARF5

### 编译选项

```
-g                             生成调试信息（DWARF5默认）
-gdwarf-4                      生成DWARF4调试信息
-gdwarf-5                      生成DWARF5调试信息
-gcodeview                     生成CodeView调试信息（Windows）
-gir-debug                     在IR中保留调试元数据
-fdebug-info-for-profiling     生成性能剖析用调试信息
```

## 2.3.10 编译器测试升级 — 差分测试与模糊测试

> 来源：09-行业前沿补充优化方案 §2.10

引入编译器行业级测试方法论：差分测试验证多后端一致性，IR模糊测试检测编译器崩溃，等价性验证确保Pass变换正确性。

### 差分测试（Differential Testing）

```cpp
namespace blocktype::testing {

/// 差分测试框架：同一输入经不同路径编译，对比结果一致性
class DifferentialTester {
public:
  /// 差分测试配置
  struct Config {
    /// 对比的编译路径
    std::vector<std::string> CompilationPaths;
    /// 对比粒度：AST级 / IR级 / 目标代码级 / 运行时级
    enum class Granularity { AST, IR, Object, Runtime };
    Granularity CompareLevel = Granularity::IR;
    /// 是否允许语义等价但文本不同（如寄存器分配差异）
    bool AllowSemanticEquivalent = true;
  };

  /// 执行差分测试
  struct DiffResult {
    bool IsEquivalent;
    std::string Difference;       /// 差异描述
    std::string Path1Output;      /// 路径1输出
    std::string Path2Output;      /// 路径2输出
  };

  DiffResult test(StringRef SourceFile, const Config& Cfg);

  /// 常用差分对比模式
  /// 1. 同一前端 → 不同后端（LLVM vs Cranelift）
  /// 2. 不同优化级别（-O0 vs -O2）
  /// 3. 不同IR Pass序列
  /// 4. 增量编译 vs 全量编译
};

} // namespace blocktype::testing
```

### IR模糊测试器

```cpp
namespace blocktype::testing {

/// IR模糊测试器：生成随机IR输入，检测编译器崩溃/断言失败
class IRFuzzer {
public:
  /// IR变异策略
  enum class MutationStrategy {

  };

  /// 执行模糊测试
```

## 2.3.11 编译器缓存架构 — 后端缓存集成

> 来源：09-行业前沿补充优化方案 §2.8

引入多层编译缓存架构，后端通过 `CompilationCacheManager` 集成缓存查询和存储，避免重复编译。缓存支持本地磁盘存储和远期远程存储。

### 缓存键与缓存条目

```cpp
namespace blocktype::cache {

/// 缓存键：编译输入的哈希摘要
struct CacheKey {
  uint64_t SourceHash;           /// 源文件内容哈希
  uint64_t OptionsHash;          /// 编译选项哈希
  uint64_t VersionHash;          /// 编译器版本哈希
  uint64_t TargetTripleHash;     /// 目标三元组哈希
  uint64_t DependencyHash;       /// 依赖文件哈希（头文件/BMI）
  uint64_t CombinedHash;         /// 综合哈希

  static CacheKey compute(StringRef Source, const CompilerOptions& Opts);
};

/// 缓存条目
struct CacheEntry {
  CacheKey Key;
  std::vector<uint8_t> IRData;          /// 序列化的IR
  std::vector<uint8_t> ObjectData;      /// 编译的目标文件
  ir::IRFormatVersion IRVersion;        /// IR格式版本
  uint64_t Timestamp;                    /// 缓存时间戳
  size_t IRSize;                         /// IR大小
  size_t ObjectSize;                     /// 目标文件大小
};

} // namespace blocktype::cache
```

### 缓存存储接口

```cpp
namespace blocktype::cache {

/// 缓存存储接口（可插拔）
class CacheStorage {
public:
  virtual ~CacheStorage() = default;

  /// 查询缓存
  virtual Optional<CacheEntry> lookup(const CacheKey& Key) = 0;

  /// 存入缓存
  virtual bool store(const CacheKey& Key, const CacheEntry& Entry) = 0;

  /// 缓存统计
  struct Stats { size_t Hits; size_t Misses; size_t Evictions; size_t TotalSize; };
  virtual Stats getStats() const = 0;
};

/// 本地磁盘缓存
class LocalDiskCache : public CacheStorage {
  std::string CacheDir;
  size_t MaxSize;  /// 最大缓存大小（字节）
public:
  explicit LocalDiskCache(StringRef Dir, size_t Max);
  Optional<CacheEntry> lookup(const CacheKey& Key) override;
  bool store(const CacheKey& Key, const CacheEntry& Entry) override;
};

/// 远程缓存接口（S3/GCS/Azure，远期实现）
class RemoteCache : public CacheStorage {
  std::string Endpoint;
  std::string Bucket;
public:
  Optional<CacheEntry> lookup(const CacheKey& Key) override;
  bool store(const CacheKey& Key, const CacheEntry& Entry) override;
};

} // namespace blocktype::cache
```

### 编译缓存管理器与后端的集成

```cpp
namespace blocktype::cache {

/// 编译缓存管理器
class CompilationCacheManager {
  std::unique_ptr<CacheStorage> Storage;
  bool Enabled = false;
public:
  void enable(StringRef CacheDir, size_t MaxSize);
  void disable();

  /// 查询IR缓存
  Optional<std::unique_ptr<ir::IRModule>> lookupIR(const CacheKey& Key,
                                                     ir::IRTypeContext& Ctx);

  /// 查询目标文件缓存
  Optional<std::vector<uint8_t>> lookupObject(const CacheKey& Key);

  /// 存入IR缓存
  bool storeIR(const CacheKey& Key, const ir::IRModule& M);

  /// 存入目标文件缓存
  bool storeObject(const CacheKey& Key, ArrayRef<uint8_t> Data);
};

} // namespace blocktype::cache
```

### 后端缓存集成流程

```
CompilerInstance::compileFile()
    ↓
1. CacheKey::compute(Source, Options)
    ↓
2. CompilationCacheManager::lookupObject(Key)
    ↓ 命中 → 直接返回缓存的目标文件
    ↓ 未命中 ↓
3. Frontend::compile() → IRModule
    ↓
4. CompilationCacheManager::lookupIR(Key)
    ↓ 命中 → 跳过前端，用缓存IR继续
    ↓ 未命中 ↓
5. CompilationCacheManager::storeIR(Key, IRModule)
    ↓
6. Backend::emitObject(IRModule) → 目标文件
    ↓
7. CompilationCacheManager::storeObject(Key, ObjectData)
    ↓
8. 返回目标文件
```

**集成要点**：
- 缓存查询在编译管线入口处执行，命中则跳过对应阶段
- IR缓存和目标文件缓存独立管理，可分别命中
- `CacheKey` 包含编译器版本哈希，版本升级自动失效
- Phase B 实现 `LocalDiskCache` 基础
- Phase E 实现 `CompilationCacheManager` 完整集成
- 远期（Phase F）实现 `RemoteCache` 远程缓存
